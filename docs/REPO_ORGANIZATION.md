# Agent Repo Organization

TStack should become two product repos plus a temporary migration workspace.

## Repo 1: `agent-registry`

Purpose: definitions, contracts, discovery.

```text
agent-registry/
  skills/
  agents/
  workflows/
  adapters/
  tool-providers/
  stacks/
  domains/
  profiles/
  schemas/
  generated/
    registry.json
  docs/
```

Rules:

- Own source-of-truth definitions.
- Own schemas for registry records.
- Own generated discovery indexes.
- Own content contracts for skills, agents, workflows, adapters, stacks, domains, profiles, and tool providers.
- Do not execute workflows.
- Do not own runtime approvals, MCP calls, audit writing, or test execution.

## Repo 2: `agent-harness`

Purpose: runtime execution.

```text
agent-harness/
  loader/
  planner/
  workflow-engine/
  policy-engine/
  approval-gates/
  mcp-client/
  adapter-runtime/
  event-bus/
  audit-log/
  test-runner/
  cli/
  sdk/
```

Rules:

- Consume `agent-registry/generated/registry.json`.
- Resolve definitions into executable runs.
- Own policy evaluation, approval gates, tool calls, events, audit logs, and test execution.
- Do not own large skill catalogs or stack/domain content.
- Do not hand-author registry definitions.

## Temporary Repo: `tstack`

Purpose: migration workspace.

Current repo keeps legacy `agent-architecture` content only until `agent-registry` can consume it without metadata loss.

Bridge command:

```bash
npm run registry:export
```

Output:

```text
generated/agent-registry/registry.json
```

That output is shaped for `agent-registry`, with sections:

```text
skills
agents
workflows
adapters
toolProviders
stacks
domains
profiles
schemas
```

## Consumption Contract

`agent-harness` should need only:

1. Registry URL or local path.
2. Registry version.
3. Definition IDs.
4. Lockfile generated by resolver.
5. Runtime credentials and policy inputs.

`agent-harness` should not need to know `agent-architecture` path conventions.

## Migration Order

1. Export legacy `agent-architecture` content to `generated/agent-registry/registry.json`.
2. Import/export same shape from `agent-registry`.
3. Make `agent-harness` load that registry shape.
4. Move definitions into `agent-registry` directories.
5. Remove `agent-architecture` as source once parity checks pass.
