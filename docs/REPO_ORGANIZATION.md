# Agent Repo Organization

TStack separates authored definitions, registry publishing, and runtime execution.

## Repo 0: `agent-pack`

Purpose: authored definitions.

```text
agent-pack/
  skills/
  agents/
  workflows/
  adapters/
  tool-providers/
  stacks/
  domains/
  profiles/
  plugins/
  generated/
  docs/
```

Rules:

- Own authored skills, agents, workflows, adapters, stacks, domains, profiles, and tool providers.
- Own templates and generated host artifacts for Claude, Codex, Copilot, and future hosts.
- Export registry-shaped definitions.
- Do not own final runtime execution.

## Repo 1: `agent-registry`

Purpose: published definitions, contracts, discovery.

```text
agent-registry/
  skills/
  agents/
  workflows/
  adapters/
  tool-providers/
  stacks/
  domains/
  profiles/
  schemas/
  generated/
    registry.json
  docs/
```

Rules:

- Own published registry records imported from `agent-pack`.
- Own schemas for registry records.
- Own generated discovery indexes.
- Own content contracts for skills, agents, workflows, adapters, stacks, domains, profiles, and tool providers.
- Do not execute workflows.
- Do not own runtime approvals, MCP calls, audit writing, or test execution.

## Repo 2: `agent-harness`

Purpose: runtime execution.

```text
agent-harness/
  loader/
  planner/
  workflow-engine/
  policy-engine/
  approval-gates/
  mcp-client/
  adapter-runtime/
  event-bus/
  audit-log/
  test-runner/
  cli/
  sdk/
```

Rules:

- Consume `agent-registry/generated/registry.json`.
- Resolve definitions into executable runs.
- Own policy evaluation, approval gates, tool calls, events, audit logs, and test execution.
- Do not own large skill catalogs or stack/domain content.
- Do not hand-author registry definitions.

## Current Repo: `tstack`

Purpose: governance workspace plus current `agent-pack` source location.

Current repo keeps the `agent-pack` source and governance tooling until
`agent-pack` moves to its own repo and split repos consume the exported registry
without metadata loss.

Bridge command:

```bash
npm run registry:export
```

Output:

```text
generated/agent-registry/registry.json
```

That output is shaped for `agent-registry`, with sections:

```text
skills
agents
workflows
adapters
toolProviders
stacks
domains
profiles
schemas
```

## Consumption Contract

`agent-harness` should need only:

1. Registry URL or local path.
2. Registry version.
3. Definition IDs.
4. Lockfile generated by resolver.
5. Runtime credentials and policy inputs.

`agent-harness` should not need to know `agent-pack` path conventions.

## Split Order

1. Export `agent-pack` content to `generated/agent-registry/registry.json`.
2. Import/export same shape from `agent-registry`.
3. Make `agent-harness` load that registry shape.
4. Move `agent-pack` to its own repo once parity checks pass.
5. Keep `tstack` as governance/docs umbrella or archive.
